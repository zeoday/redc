# 自定义部署增强功能设计文档

## 概述

本文档记录了自定义部署功能的两次重要增强：
1. 火山引擎抢占式实例支持
2. AI 错误分析功能

---

## 1. 火山引擎抢占式实例支持

### 1.1 问题背景

用户希望在自定义部署时能够使用抢占式实例（Spot Instance）以降低成本。火山引擎的抢占式实例通过 `instance_charge_type` 和 `spot_strategy` 参数配置。

### 1.2 技术实现

#### 1.2.1 模板变量定义

**文件**: `redc-template/base-templates/volcengine-ecs/variables.tf`

```hcl
variable "is_spot_instance" {
  description = "是否为抢占式实例 (火山引擎暂不支持，此参数无效)"
  type        = bool
  default     = false
}
```

#### 1.2.2 Terraform 资源配置

**文件**: `redc-template/base-templates/volcengine-ecs/main.tf`

```hcl
resource "volcengine_ecs_instance" "this" {
  # ... 其他配置 ...
  
  # 计费类型：按量付费
  instance_charge_type = "PostPaid"
  
  # 抢占式实例策略
  # - SpotAsPriceGo: 系统自动出价，跟随市场价格
  # - NoSpot: 不使用抢占式
  spot_strategy = var.is_spot_instance ? "SpotAsPriceGo" : "NoSpot"
}
```

### 1.3 关键要点

1. **火山引擎 API 限制**: `instance_charge_type` 只支持 `PostPaid`（按量付费）和 `PrePaid`（包年包月），不支持直接的 "Spot" 值

2. **抢占式实例配置方式**: 需要同时设置：
   - `instance_charge_type = "PostPaid"` (按量付费)
   - `spot_strategy = "SpotAsPriceGo"` (跟随市场价格的抢占式实例)

3. **spot_strategy 有效值**:
   - `NoSpot` - 不使用抢占式
   - `SpotAsPriceGo` - 系统自动出价，跟随市场价格
   - `SpotWithPriceLimit` - 设置价格上限的抢占式

### 1.4 错误处理

#### 错误: "expected instance_charge_type to be one of [PostPaid PrePaid], got Spot"

**原因**: 直接使用 `instance_charge_type = "Spot"` 

**解决方案**: 使用 `instance_charge_type = "PostPaid"` + `spot_strategy = "SpotAsPriceGo"` 组合

#### 错误: "expected spot_strategy to be one of [NoSpot SpotAsPriceGo SpotWithPriceLimit], got"

**原因**: `spot_strategy` 设置为空字符串

**解决方案**: 使用 `NoSpot` 作为非抢占式实例的值

## 2. AI 错误分析功能

### 2.1 功能概述

当自定义部署失败时，用户可以通过点击 "AI 分析错误原因" 按钮，让 AI 自动分析错误信息并提供解决方案。

### 2.2 技术架构

#### 2.2.1 后端实现

**文件**: `app.go`

```go
// AnalyzeDeploymentError uses AI to analyze deployment errors and provide solutions
func (a *App) AnalyzeDeploymentError(deploymentID, errorMessage, provider, templateName string) error {
    // 1. 获取 AI 配置
    profile, err := redc.GetActiveProfile()
    if err != nil || profile.AIConfig == nil {
        return fmt.Errorf("请先配置 AI 服务")
    }
    
    // 2. 创建 AI 客户端（复用现有 AI 模块）
    client := ai.NewClient(aiConfig.Provider, aiConfig.APIKey, aiConfig.BaseURL, aiConfig.Model)
    
    // 3. 构建 prompt
    systemPrompt := `你是一个云资源部署专家助手...`
    
    // 4. 流式调用 AI API
    err = client.ChatStream(ctx, messages, func(chunk string) error {
        // 5. 发送事件到前端
        runtime.EventsEmit(a.ctx, "ai-deployment-error-chunk", map[string]string{
            "deploymentId": deploymentID,
            "chunk":        chunk,
        })
        return nil
    })
    
    // 6. 发送完成事件
    runtime.EventsEmit(a.ctx, "ai-deployment-error-complete", map[string]interface{}{
        "deploymentId": deploymentID,
        "success":      true,
    })
    
    return nil
}
```

#### 2.2.2 前端实现

**文件**: `frontend/src/components/CustomDeployment/CustomDeploymentList.svelte`

```svelte
<script>
  // AI 分析状态
  let aiAnalyzing = $state<Record<string, boolean>>({});
  let aiAnalysisResult = $state<Record<string, string>>({});
  let aiAnalysisCompleted = $state<Record<string, boolean>>({});
  
  // 监听 AI 返回的 chunk
  EventsOn('ai-deployment-error-chunk', (data) => {
    const { deploymentId, chunk } = data;
    if (deploymentId && chunk) {
      aiAnalysisResult[deploymentId] = (aiAnalysisResult[deploymentId] || '') + chunk;
      aiAnalysisResult = { ...aiAnalysisResult };
    }
  });
  
  // 监听 AI 完成事件
  EventsOn('ai-deployment-error-complete', (data) => {
    const { deploymentId, success } = data;
    if (deploymentId) {
      aiAnalyzing[deploymentId] = false;
      aiAnalysisCompleted[deploymentId] = true;
      // 触发响应式更新
      aiAnalyzing = { ...aiAnalyzing };
      aiAnalysisCompleted = { ...aiAnalysisCompleted };
    }
  });
  
  // 触发 AI 分析
  async function handleAIAnalysis(deploymentId, errorMessage, provider, templateName) {
    // 检查 AI 配置
    const profile = await GetActiveProfile();
    if (!profile?.aiConfig?.apiKey) {
      alert('请先在设置中配置 AI 服务');
      return;
    }
    
    // 开始分析
    aiAnalyzing[deploymentId] = true;
    aiAnalysisResult[deploymentId] = '';
    aiAnalyzing = { ...aiAnalyzing };
    aiAnalysisResult = { ...aiAnalysisResult };
    
    try {
      await AnalyzeDeploymentError(deploymentId, errorMessage, provider, templateName);
    } catch (err) {
      alert(`AI 分析失败: ${err.message}`);
      aiAnalyzing[deploymentId] = false;
      aiAnalyzing = { ...aiAnalyzing };
    }
  }
</script>
```

### 2.3 事件机制

| 事件名 | 方向 | 数据格式 | 说明 |
|--------|------|----------|------|
| `ai-deployment-error-chunk` | 后端 → 前端 | `{deploymentId, chunk}` | AI 返回的文本片段 |
| `ai-deployment-error-complete` | 后端 → 前端 | `{deploymentId, success}` | AI 分析完成 |

### 2.4 关键要点

1. **事件命名**: 事件名不要加 deploymentID 后缀，否则前端难以统一监听

2. **数据格式**: 发送事件时带上 deploymentId，以便前端区分不同部署的分析结果

3. **状态管理**: 需要三个状态：
   - `aiAnalyzing`: 是否正在分析中（显示加载动画）
   - `aiAnalysisResult`: 分析结果内容
   - `aiAnalysisCompleted`: 分析是否已完成（完成后仍显示结果）

4. **Svelte 5 响应式**: 修改状态后需要使用 `state = { ...state }` 触发响应式更新

## 3. 配置凭证获取优化

### 3.1 问题背景

原代码只从环境变量获取云厂商凭证，但用户通过 GUI 配置的凭证保存在配置文件中。

### 3.2 解决方案

#### 3.2.1 添加全局配置变量

**文件**: `mod/config.go`

```go
var LoadedConfig *Config = nil // 全局配置对象

// 在 LoadConfig 和 SaveConfig 中更新
LoadedConfig = &conf
```

#### 3.2.2 添加凭据获取函数

```go
// GetProviderCredentials 从已加载的配置中获取云厂商凭据
func GetProviderCredentials(provider string) (accessKey, secretKey string) {
    if LoadedConfig == nil {
        return "", ""
    }
    
    switch provider {
    case "volcengine":
        return LoadedConfig.Providers.Volcengine.AccessKey, LoadedConfig.Providers.Volcengine.SecretKey
    // ... 其他云厂商
    }
    
    return "", ""
}
```

#### 3.2.3 优先级

1. 配置文件 (`~/redc/config.yaml`) - 优先
2. 环境变量 - 备用

## 4. Terraform 错误信息优化

### 4.1 问题

Terraform 执行失败时，错误信息不够详细，用户无法了解具体原因。

### 4.2 解决方案

#### 4.2.1 捕获 stderr 输出

**文件**: `mod/terraform.go`

```go
// stderrWriter 捕获 stderr 输出的 writer
type stderrWriter struct {
    buf *strings.Builder
}

func (w *stderrWriter) Write(p []byte) (int, error) {
    w.buf.WriteString(string(p))
    return len(p), nil
}

// 在 TerraformExecutor 中使用
te := &TerraformExecutor{
    // ...
    stderr: &stderrWriter{buf: &strings.Builder{}},
}

tf.SetStderr(te.stderr)
```

#### 4.2.2 错误返回优化

```go
func (te *TerraformExecutor) Apply(ctx context.Context, opts ...tfexec.ApplyOption) error {
    err := te.tf.Apply(ctx, opts...)
    if err != nil {
        var stderrMsg string
        if sw, ok := te.stderr.(*stderrWriter); ok {
            stderrMsg = strings.TrimSpace(sw.buf.String())
        }
        // 只有当 stderr 有内容且原始错误较简短时才追加
        if stderrMsg != "" && len(err.Error()) < 100 {
            return fmt.Errorf("%w\n\n%v", err, stderrMsg)
        }
        return err
    }
    return nil
}
```

#### 4.2.3 保存错误到数据库

**文件**: `mod/custom_deployment.go`

```go
if err := TfApply(deploymentPath); err != nil {
    // 将错误信息保存到 outputs 中
    errorMsg := err.Error()
    deployment.Outputs = map[string]interface{}{
        "error_message": errorMsg,
    }
    
    // 更新状态为错误
    deployment.State = StateError
    deployment.UpdatedAt = time.Now()
    deployment.DBSave()
    
    return fmt.Errorf("Terraform apply 失败: %w", err)
}
```

## 6. 注意事项

1. **Terraform Provider 限制**: 不同云厂商的 Terraform Provider 对抢占式实例的支持程度不同，需要针对每个厂商单独测试

2. **AI 配置依赖**: AI 分析功能依赖用户在设置中配置 AI 服务（API Key、Base URL、Model）

3. **错误信息格式化**: 存储到数据库的错误信息需要考虑长度限制和格式兼容性
